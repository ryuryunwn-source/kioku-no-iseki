## 0. ゲームの売り（独自性のポイント）

1. **デッキを共有する** — 各プレイヤーは自分専用のデッキを持たず、中央の「遺構デッキ」を2人で掘り合う。
2. **マナは減衰する** — 使わずに溜め込むと記憶が薄れていき、資源が自然に減っていく。
3. **死んだカードが共有デッキに戻る** — 破壊されたユニットは自分の墓地に行かず、刻印（強化痕）を保持したまま共有デッキに戻る。相手がそれを掘り当てることもある。
4. **【v2予定】自分の写真がそのままカードになる** — 対戦前に自分のカメラロールから写真を選ぶと、その写真を元にステータス・技・名前が自動生成される。同じ写真は常に同じカードになる（ガチャ的な運要素ではなく、写真選びそのものが戦略になる）。
5. **【v2予定】技は声で発動する（真名詠唱・PC版限定）** — 各カードには固有の「真名」がある。手札やボード上のカードを操作するのではなく、その名前を声に出して呼ぶことで技が発動する。v1では全プラットフォームでタップ発動のみ（仕組み自体はv1から存在し、v2は入力方法を1つ追加するだけ）。

## 1. 世界観（フレーバーテキスト）

太古、星を渡る知識を持った超古代文明が存在した。その文明は一夜にして滅び、知識はすべて「記憶石板（メモリー・タブレット）」と呼ばれる遺物に封じられた。

プレイヤーは、その文明の生まれ変わりである**「想起者（リコーラー）」**。完全な記憶は持たないが、戦場に散らばる記憶石板に触れることで、前世の力を一時的に「想起」できる。

この文明には文字がなく、すべての知識は**声と名**によって受け継がれた。力ある記憶には必ず「真名」が宿り、それを正しく声に出して呼んだ者だけが、その力を引き出すことができる――黙って念じるだけでは何も起こらない。

そして現世の想起者たちは、自らの今生の記憶（写真）を遺構に捧げることで、新たな「写し身」を生み出す。写し身は、捧げた本人だけのものではない。遺構に沈んだ記憶は、いつか敵の手によって掘り出され、その真名を呼ばれることもある。

## 5. セットアップ

1. 固定カード48枚（守護者30・想起術14・礎石4）をよくシャッフルし、中央に1つだけ配置する。
2. 各プレイヤーは遺構デッキの上から5枚を発掘し、初期手札とする。
3. 先手・後手を決める。後手プレイヤーは追加で1枚発掘してよい。
4. HPを20、想起ゲージを2/2にセットする。

> 【v2予定】写真機能を追加する際は、この前段に「写真選択→カード生成・確認」のステップが入り、守護者30枚の一部または全部が各プレイヤーの写し身に置き換わる（15章）。v1ではこのステップは存在しない。

## 6. ターンの流れ（フェイズ構成）

1. **想起減衰フェイズ**（自動）
2. **発掘フェイズ**（1枚発掘）
3. **刻むフェイズ**（任意・1ターン1回まで）
4. **行動フェイズ**（カード使用・攻撃・詠唱、メイン）
5. **終了フェイズ**（風化カウンター処理）

実装上はこの5フェイズをそのままステートマシンの状態として扱うのが綺麗です（`enum TurnPhase { Decay, Excavate, Inscribe, Action, End }`）。

---

## 7. フェイズ詳細

### 7-1. 想起減衰フェイズ
- 自分の想起ゲージを **-1** する（最低0）。
- 例外：前のターンに「刻む」を行った場合、このターンの減衰は発生しない。

### 7-2. 発掘フェイズ
- 遺構デッキの一番上を1枚発掘し、手札に加える。
- 遺構デッキが0枚になった場合：両者の記憶領域のカード枚数を比較し、多い方が即勝利（同数ならHPが高い方）。

### 7-3. 刻むフェイズ（任意行動）
- 手札を1枚「捧げる」（ゲームから除外する）ことで、想起ゲージの上限を+1し、即座に上限まで全回復する。
- 1ターンに1回のみ。この行動を行うと次の想起減衰フェイズはスキップされる。

### 7-4. 行動フェイズ
- 守護者・想起術・礎石をプレイ可能（コストは想起ゲージから支払う）。
- 場の守護者は攻撃宣言ができる（召喚酔いあり＝出した直後の番は攻撃不可）。
- **このフェイズ中、自分の場にいる守護者をタップすると、その技が発動できる**（16章）。発動は1ターンにつき同じ守護者1体から1回まで。通常の攻撃・カードプレイと組み合わせて自由な順番で行ってよい。

### 7-5. 終了フェイズ
- 3ターン以上手札に留まったカードは風化カウンターが1増える。
- 風化カウンターが3に達したカードは消滅し、瓦礫トークン1個になる。

## 8. カードタイプ詳細

### 守護者（Guardian）
- パラメータ：コスト／攻撃力／防御力／系統／刻印数（初期値0）／**真名**／**技**／**守護(一部)**
- 場に出すと召喚酔い状態（次の自分の番から攻撃可能）。技の発動（タップ）は召喚酔い中でも可能。

#### 守護（Guard）キーワード
- 一部の守護者だけが持つ。**相手の場に守護持ちがいる間、その相手の本体(HP)を直接攻撃できない**（守護を全部倒してから本体へ）。守護以外の守護者は通常どおり攻撃できる。
- 守護は**召喚酔い中でも機能**（防御は受けなので酔い無関係）。除去・全体ダメージで処理できる。
- v1で守護を持つ6体：苔むす守り手(森)・森林の巨人(森)・氷河の守護者(流)・聖なる番兵(光)・神域の守護騎士(光)・天啓の大司祭(光)。焔・影は純アタッカー（守護なし）。
- 【v2予定】写真から自動生成される「写し身」も守護者の一種として追加され、固定の守護者と完全に同じルールで扱われる（攻撃・破壊・転生すべて共通）。v1ではすべての守護者が固定デザインのみ。

### 想起術（Recollection）
- パラメータ：コスト／効果テキスト／使用条件
- 使用後は遺構デッキの一番下に戻る。

### 礎石（Cornerstone）
- パラメータ：コスト／永続効果／設置上限（基本1スロット）
- 破壊されると遺構デッキの下に戻る（刻印は持たない）。

---

## 9. 風化ルール

- 風化カウンターは手札の中でのみ進行する。
- カードを手札から出して戻す効果がある場合、風化カウンターは0にリセットされる。
- 風化で消滅したカードは遺構デッキにも戻らず、完全にゲームから除外される。

---

## 10. 転生ルール

1. 守護者が破壊された場合、即座に遺構デッキのランダムな位置にシャッフルして戻す。
2. その守護者カードに刻印を1つ刻む。
3. 刻印を持つ守護者は再発掘・再登場時、刻印1つにつき攻撃力+1／防御力+1のボーナスを得る。
4. 真名・技・系統はカードに紐づく不変データであり、誰が掘り当てても変わらない。つまり相手が自分の守護者を掘り当てた場合、相手はその技を使うことができる。
5. 刻印が3つ蓄積した守護者が再度破壊された場合、破壊した側の記憶領域に直接送られる。

---

## 11. 勝利条件

- **通常勝利**：相手のHPを0以下にする。
- **古き盟約勝利**：自分の記憶領域に完全刻印（刻印3つ）の守護者が3体集まった時点で即勝利。
- **遺構枯渇勝利**：遺構デッキが尽きた際、記憶領域のカード枚数が多い方が勝利。

## 12. 裁定・エッジケース

| ケース | 裁定 |
|---|---|
| 風化で消滅したカードに刻印がある場合 | 刻印情報は破棄され、完全にゲームから除外（復活なし） |
| 同じターンに両プレイヤーが同時に守護者を破壊し合った（相討ち） | 手番プレイヤーのカードを先に処理 |
| 刻むフェイズで捧げる手札が0枚 | 刻む行動自体を選択不可にする |
| 礎石の効果が複数重複する | 加算（乗算ではない）で処理する |
| **オンライン対戦中に相手が切断した** | 対戦を終了し、「対戦相手との接続が切れました」と表示する（v1では再接続非対応） |
| **オンライン対戦で、クライアント側から不正なリクエスト（自分の手札にないカードを出す等）が来た** | ホスト側のServerRpc内で必ず検証し、不正なら無視する（クライアントの言い分をそのまま信用しない） |

【v2予定・写真生成と音声を実装する際のエッジケース】

| ケース | 裁定 |
|---|---|
| 音声認識が真名を正しく聞き取れなかった | 発動失敗とし、ペナルティなし・行動権も消費しない。何度でも言い直せる。タップでの代替発動も常に可能 |
| 認識結果が真名と微妙に違う（かな表記のブレ） | 簡易ファジーマッチ（編集距離1〜2程度）で許容判定する |
| 相手の番に自分の写し身の真名を口に出してしまった | 何も起こらない（発動は「現在の手番プレイヤーの守護者」かつ「自分の行動フェイズ中」のみ有効） |
| 生成された真名が同じ対戦内で重複した | 15章の生成ロジックに従い、衝突した側を次の候補名に自動でずらす（生成時に解決、対戦中は常に一意） |
| 選んだ写真が極端に単色・破損している等で生成に失敗する | フォールバック値（系統:影、最低保証ステータス）を割り当て、生成は必ず成功させる |
| AI呼び出しが失敗する・インターネット接続がない（カード生成時） | 15-6のフォールバック（固定候補リストからの機械的選択）に自動で切り替える。ゲームが進行不能になることはない |
| AIが返した真名が条件（カタカナ2〜4モーラ）を満たさない | 15-6のバリデーションで補正する。補正後も不正な場合はフォールバック候補リストから選ぶ |

## 14. アート・UI仕様

### 14-1. 設計方針：「枠」と「イラスト」を分離する

**カードの見た目を「枠（フレーム）」と「イラスト」の2層に分け、別々に作る。** これは過去のプロジェクトで「テクスチャやUIの修正が多かった」反省を踏まえた設計判断。

- **枠（フレーム）**：カード名・コスト・攻撃力／防御力・系統アイコン・技テキストなどを表示するUI部分。Unity側で**1セットだけ作る固定テンプレート**（系統ごとに5色のバリエーションのみ）。48枚すべてがこの同じテンプレートに乗る。
- **イラスト**：枠の中の絵窓（正方形〜縦長の決まったサイズの領域）に差し込まれる1枚絵。30種の守護者ごとに別の画像になる。

枠を固定テンプレートにすることで、**レイアウト崩れ・文字の見切れ・配色の不統一といったUI側の問題は、イラストの出来とは無関係に防げる。** イラストはAI生成で個体差が出ても、枠が統一されていれば「同じゲームのカード」に見える（実在のトレーディングカードゲームも、絵師が1枚ごとに違うのに枠が統一されているので破綻しない）。

### 14-2. イラスト生成方針（AI画像生成）

v1からAI画像生成（Midjourney等）でイラストを用意する。30枚で画風が散らないよう、以下の手順を踏む。

1. **アンカー画像を先に1〜2枚作る**：狙う画風（例：「古代遺跡・風化した石板の質感、落ち着いた色調、写実とイラストの中間」）を言葉とアンカー画像の両方で固定する。
2. **Style Reference機能を全カードで使い回す**：Midjourneyの`--sref`（Style Reference）にアンカー画像を指定し、30種すべてのプロンプトに同じ`--sref`を付ける。`--sref`は色味・画材感・質感・ライティングだけを引っ張る機能で、キャラクターそのものを固定する機能（`--cref`）とは別物。系統ごとに違う守護者を描いても画風だけは統一できる。
3. **1セッションでまとめて生成する**：日をまたぐと自分の好みやモデルのバージョンがブレやすいので、可能な限り一度に30枚分を回す。
4. **最後に並べて確認する**：30枚のサムネイルを一覧表示し、明らかに浮いているものだけ`--sref`の重みやプロンプトを調整して再生成する。

### 14-3. Unity側の技術仕様（Claude Codeへの指示用）

- イラスト画像は `Assets/Resources/CardArt/` 以下に、`guardian_001.png`〜`guardian_030.png` のように13章の守護者リストの並び順に対応するファイル名で配置する想定。
- 推奨解像度：512×512px（正方形）、PNG。Unityのテクスチャインポート設定は `Sprite (2D and UI)` ・ Compression は `Normal Quality` を基本にする。
- カードの枠（フレーム）はUI側のPrefab/Imageコンポーネントとして1種類だけ作り、系統（焔/森/流/光/影）ごとに枠の縁色だけ変える5バリエーションとする。イラストはそのPrefab内の決まった矩形領域（イラスト窓）にハメ込む形にする。
- 想起術・礎石カードにもイラストを付けるかはv1では未確定。**ひとまず守護者30種のみイラストを用意し、想起術・礎石は枠とアイコンのみのシンプルな見た目で進める**（必要なら後から追加できるよう、枠側の設計に想起術・礎石用のイラスト窓も予約だけしておく）。

### 14-4. UIレイアウトの確認プロセス（運用ルール）

イラスト・枠の統一だけでは「UIのレイアウト崩れ・操作性」の問題は防げない。これは別途、進め方で対処する。

- Claude Codeに画面をまとめて作らせず、**1画面（例：対戦画面のレイアウトのみ）を作った時点でスクリーンショットを見せてもらい、レイアウトを承認してから次の画面に進む**運用にする。
- 承認の単位の例：①対戦画面全体のレイアウト（手札・場・HP/ゲージの配置）→②カード1枚のUI（枠＋イラスト窓＋テキスト）→③技発動時の演出→④AI対戦・オンライン対戦の専用UI（join code表示画面など）。
- 1機能ぶんまるごと実装してから初めて見た目を確認する、という進め方は避ける。

## 15. 【v2予定】写し身カードの自動生成ロジック（実装仕様）

> **この章はv1では実装しない。** v1の守護者は13章の固定30種のみを使う。この章は、将来「画像挿入」「画像の自動命名」機能を追加する際に、既存のv1コード（`CardInstance`が`CardDefinition`と`GeneratedGuardianData`の両方を受け入れる設計になっている、19章参照）に大きな変更を入れずに差し込めるよう、あらかじめ仕様を決めておくためのものである。

**設計方針：強さ（コスト・攻撃力・防御力・系統・技の効果内容）は完全に決定論的なアルゴリズム（ハッシュ計算）で決める。名前・技名などの「フレーバー」のみ、写真の内容を実際に見たAIが生成する。** この2層に分けることで、バランス管理はアルゴリズム側で保証しつつ、写真ごとの個性はAIの命名で表現する。

### 15-1. 生成パイプライン

```
入力: 画像ファイル（Texture2D）
  ↓
① 画像を8x8にダウンサンプリングし、平均色(R,G,B)と明度・彩度を算出
② 画像バイト列から決定論的ハッシュ値(64bit)を計算（例: FNV-1aハッシュ）
  ↓
③ 系統(Element)を平均色のHSVから判定 ※強さ・系統はここまで全てアルゴリズムのみ
④ ハッシュ値の一部を使って「予算ティア」を決定（低/中/高の3段階）
⑤ 明度から攻撃力/防御力の配分比を決定
⑥ ハッシュ値の別の部分で技テンプレートを選択し、予算ティアで効果量をスケール
  ↓
⑦ キャッシュ確認：このハッシュ値の命名結果が保存済みか？
    YES → 保存済みの真名・技名を使う（API呼び出しなし）
    NO  → ⑧へ
⑧ 画像をAI（Vision対応モデル）に渡し、真名・技名を生成させる（15-6参照）
⑨ 生成結果をハッシュ値に紐づけてローカルに保存（以後はキャッシュから取得）
  ↓
出力: { TrueName, Element, Cost, Attack, Defense, Technique }
```

**重要：この処理は対戦中ではなく、5章の「カード生成・確認」ステップ（対戦開始前）で1回だけ行う。** 対戦中に音声でカードを呼ぶ処理（16章）とは完全に別のタイミングなので、API呼び出しの数秒の待ち時間が対戦のテンポに影響することはない。

### 15-2. 系統（Element）判定

- 彩度が低い（くすんだ/モノクロ系）場合：明度が高い→**光**、明度が低い→**影**
- それ以外はHueで判定：暖色(赤〜橙)→**焔** / 黄緑〜緑(50°〜160°)→**森** / 青緑〜青(160°〜260°)→**流** / それ以外(紫〜赤紫)→**影**

### 15-3. ステータス配分

- 予算ティア：`hash % 3` → 低(コスト2・攻防合計6)／中(コスト3・攻防合計8)／高(コスト4・攻防合計10)
- 攻撃比率：明度を0.3〜0.7にクランプして使用（明るい写真ほど攻撃寄り）
- `攻撃力 = round(攻防合計 × 攻撃比率)`、`防御力 = 攻防合計 - 攻撃力`

### 15-4. 技テンプレート

系統ごとに4種類程度の技テンプレートを用意し、`(hash >> 8) % テンプレート数` で選択。効果量は予算ティアでスケール（低=弱め、高=強め）。**全テンプレートは対象選択を必要としない自己解決型**にする（声だけで完結させるため）。

| テンプレート分類 | 効果の型 | デフォルト対象 |
|---|---|---|
| 単体攻撃 | 固定ダメージ | 相手の防御力最小の守護者（いなければ相手に直接） |
| 範囲攻撃 | 全体に少量ダメージ | 相手の場の守護者全体 |
| 自己強化 | 攻撃力or防御力アップ | 発動した写し身自身 |
| 弱体 | 攻撃力ダウン | 相手の場の攻撃力最大の守護者 |
| 回復 | HPまたは想起ゲージ回復 | 自分自身 |

### 15-5. 真名・技名の生成（AIビジョン方式）

**真名と技名は、写真の内容を実際に見たAIが生成する。** 色やハッシュだけで決める方式（旧版）はオフライン時のフォールバックとしてのみ残す。

- 生成されるのは「真名」（カード自体の名前・声で呼ぶ対象）と「技名」（その写し身が持つ技のフレーバー名。例：「若葉の守り」）の2つ。**効果内容・対象・数値は15-4のテンプレートに従うアルゴリズム側の管轄であり、AIはそこに関与しない。**
- 真名は音声詠唱（16章）の対象になるため、**カタカナ・2〜4モーラ・発音しやすい**という制約をAIへのプロンプトで明示する。
- **同じ写真には常に同じ名前を使う（決定論性の保証）**：画像ハッシュ→生成結果のマッピングをローカルに永続保存（キャッシュ）し、同じ写真が再度使われた場合はAIを呼ばずキャッシュを返す。これにより「写真選びが戦略になる」という性質と「APIコストが対戦ごとに毎回かかる」問題の両方を解決する。

### 15-6. AI呼び出しの実装仕様

```
リクエスト: 画像(base64) + 以下のような指示プロンプトをAPIに送信
  「この画像を見て、古代文明・忘れられた記憶をテーマにしたカードゲームのための
   名前を考えてください。以下のJSON形式のみで出力してください（説明文不要）：
   { "trueName": "カタカナ2〜4モーラの名前", "techniqueName": "技の短い名前" }」

レスポンス: 上記JSONをパースして trueName / techniqueName を取得
  ↓
バリデーション:
  - trueNameがカタカナのみ・2〜4モーラか？ → NOなら範囲外の場合は後ろを切り詰める/前方一致部分のみ使う
  - 同一対戦内で真名が重複していないか？ → 重複していたら技名はそのまま、真名の末尾に
    区別用の音を1つ追加する等のルールで一意化する
  ↓
キャッシュに保存: { imageHash: { trueName, techniqueName } }
```

**フォールバック（API呼び出しに失敗した場合・オフライン時）**：旧15-5方式に切り替える。事前に用意した発音しやすいカタカナ名150〜200個の候補リスト（`TrueNameCandidates.asset`）から `index = imageHash % candidateList.Count` で1つを機械的に選ぶ。技名も同様に系統ごとの固定候補リストから選ぶ。これにより、**インターネット接続がない場合でもゲーム自体は必ず動作する**（名前の個性は失われるが、対戦は成立する）。

### 15-7. 実務上の注意（ゲーム公開時に再検討が必要な点）

- AI呼び出しにはAnthropic APIキーが必要。**個人で遊ぶ分には問題ないが、もし将来このゲームを他の人に配布する場合、APIキーをアプリ内に直接埋め込むのは避けるべき**（誰でも取り出せてしまい、悪用されると課金が発生する）。配布する場合は、APIキーを保持する簡易サーバー（プロキシ）を別途用意し、アプリはそのサーバー経由でリクエストする構成にするのが安全。今は個人利用なので直接埋め込みでも進められるが、配布段階になったらこの点をClaude Codeと再度相談すること。
- 15枚の写真を選ぶたびに（未キャッシュの場合）最大15回のAPI呼び出しが発生する。並行して呼び出す（逐次待ちにしない）ことで、確認画面での待ち時間を短縮できる。

## 16. 技の発動ルール（実装仕様）

### 16-1. プレイヤー向けルール（v1）
- 自分の行動フェイズ中、自分の場にいる守護者をタップすると、その技が発動候補になる。
- 発動コスト（技ごとに固定、目安1〜2）を想起ゲージから支払えれば発動する。ゲージが足りなければ発動しない（ペナルティなし）。
- 1体の守護者が1ターンに発動できる回数は1回まで。

### 16-2. アーキテクチャ方針（v1）

public static class TechniqueActivator {

    public static bool TryActivate(string trueName, RecallerState owner, TurnPhase currentPhase) {

### 16-3. データフロー（v1）
```
カードタップ → TechniqueActivator.TryActivate(タップされたカードの真名, ...)

TryActivate内の判定:
  1. 指定された真名 = 自分の場にいる守護者の真名か？ → NOなら失敗
  2. 現在、自分の行動フェイズか？ → NOなら失敗
  3. その守護者は今ターン未発動か？ → NOなら失敗
  4. 想起ゲージ ≥ 技の発動コスト か？ → NOなら失敗（UI上で「ゲージ不足」を一瞬表示）
  5. すべて満たせば技を発動し、デフォルト対象（15-4参照）に効果を適用
```

### 16-5. 【v2予定】音声入力の追加方法

> **この節はv1では実装しない。** v1のTechniqueActivatorを変更せずに音声入力を追加する手順をまとめておく。

## 17. AI対戦モード（実装仕様）

### 17-1. 設計方針

**AIは「もう1つの入力方法」として実装する。** 声・タップが`TechniqueActivator.TryActivate`を呼ぶのと同じように、AIも既存の行動メソッド（カードを出す・攻撃する・刻む・技を発動する）をそのまま呼ぶだけにする。AI専用の特別な裏ルートやゲーム状態は作らない。

```csharp
public interface IPlayerController {
    Task TakeTurnAsync(RecallerState self, RecallerState opponent, RuinsDeck deck, GameActions actions);
}
public class HumanController : IPlayerController { /* UI入力（タップ/声）からactionsを呼ぶだけ */ }
public class AIController : IPlayerController { /* 17-3の優先順位ロジックでactionsを呼ぶ */ }

// 既存の行動メソッド群。Human/AIどちらも同じこれを呼ぶ
public class GameActions {
    public bool PlayCard(CardInstance card) { throw new System.NotImplementedException(); }
    public bool Attack(CardInstance attacker, CardInstance target /* nullなら直接攻撃 */) { throw new System.NotImplementedException(); }
    public bool Inscribe(CardInstance sacrificed) { throw new System.NotImplementedException(); }
    public bool ActivateTechnique(string trueName) { throw new System.NotImplementedException(); } // TechniqueActivatorを内部で呼ぶ
}
```

### 17-2. AIが使うカード

v1ではAIも人間と同じ固定48枚デッキ（13章）から発掘する。AI専用のカードプールは不要。AIが発掘した守護者も人間が発掘した場合と全く同じルール（風化・転生・刻印）に従う。

> 【v2予定】写真機能を追加すると、人間プレイヤーの守護者は写し身に置き換わるが、AIには「自分のカメラロール」がない。その場合は開発側で汎用の写真セットをアプリに同梱し、AI専用の写し身プールとして使う想定（ビルド時に1回だけAI命名を実行してキャッシュしておけば対戦ごとのAPI呼び出しは不要）。

### 17-3. 行動優先順位（ヒューリスティック）

複雑な探索AIではなく、優先順位リストに従う方式を推奨する（実装コストと十分な手応えのバランスが良い）。プレイテストで重みは調整する前提の初期値。

```
■ 刻むフェイズ
- 手札に「今のゲージでは支払えないコストのカード」が2枚以上あり、かつ
  捧げても惜しくないカード（最もコストが低い）がある場合 → 刻む

■ 行動フェイズ（以下を順に検討し、可能な行動があれば実行し、次のループへ）
1. 攻撃でゲームを終わらせられるなら（場の攻撃力合計 ≥ 相手HP）、全力で攻撃する
2. 発動可能な技があり、有効な対象（15-4のデフォルト対象）が盤面に存在するなら発動する
3. 手札の守護者で、今出せる中で最もコストが高いものを場に出す（スロットが空いていれば）
4. 除去系の想起術（断たれた絆など）が使えて、相手の場に最も攻撃力の高い守護者がいるなら使う
5. 上記に当てはまらない想起術・礎石を、コストが高い順に出せるだけ出す
6. 召喚酔いでない自分の守護者で、まだ攻撃していないものを全て攻撃させる
   - 相手の守護者で防御力が自分の攻撃力以下のものがあれば、それを優先して攻撃（撃破できるため）
   - なければ相手に直接攻撃する
```

### 17-4. 難易度（将来の拡張）

v1ではこの1種類のAIのみを実装する。将来的に難易度を分けたい場合は、17-3の優先順位の一部をランダムに無視させる（簡単）／先読みの深さを増やす（難しい）といった調整で対応できる。

## 18. オンライン対戦（Unity Relay + Netcode for GameObjects）

### 18-1. 設計方針（最重要）

**ゲームの正規の状態（遺構デッキの内容・両者の手札の中身）は、ホスト側だけが保持する。** 各クライアントには「見えてよい情報だけ」を送る。これを徹底しないと、クライアント側のメモリやネットワーク通信を覗けば相手の手札や山札の並びが分かってしまう（チート対策として重要）。

- 自分の手札：自分のクライアントにのみ送る
- 相手の手札：**枚数のみ**を送る（内容は送らない）
- 場（守護者・礎石）：両者の場は表向きなので、両クライアントに送ってよい
- 遺構デッキ：**残り枚数のみ**を送る（内容・並び順は送らない）

### 18-2. 採用パターン：ホスト権威モデル

ローカル対人戦・AI対戦と同じ`GameActions`（17-1参照）を、**常にホスト側のインスタンスでのみ実行する。** クライアント（参加者側）は直接`GameActions`を呼ばず、「やりたい操作」をServerRpcでホストに送り、ホストが検証・実行した結果をクライアントに返す。

```csharp
public class NetworkGameController : NetworkBehaviour {
    // ホスト側にのみ存在する正規の状態
    private RecallerState hostState;
    private RecallerState clientState;
    private RuinsDeck deck;
    private GameActions actions; // 17-1のGameActionsをホスト上でラップする

    [Rpc(SendTo.Server)]
    public void RequestPlayCardRpc(int cardInstanceId, RpcParams rpcParams = default) {
        // 送信元クライアントの手札か検証 → actions.PlayCard を実行 → 結果を両クライアントに反映
    }
    [Rpc(SendTo.Server)]
    public void RequestAttackRpc(int attackerId, int targetId) { /* 同様 */ }
    [Rpc(SendTo.Server)]
    public void RequestInscribeRpc(int sacrificedCardId) { /* 同様 */ }
    [Rpc(SendTo.Server)]
    public void RequestActivateTechniqueRpc(string trueName) { /* 同様。PC版の声もここに繋がる */ }

    // 結果の反映：相手の手札は枚数だけ、自分の手札は中身まで、対象クライアントだけに送る
    [Rpc(SendTo.SpecifiedInParams)]
    private void SyncOwnHandRpc(CardSnapshot[] hand, RpcParams rpcParams) { /* 本人のクライアントのみに送信 */ }
    [Rpc(SendTo.ClientsAndHost)]
    private void SyncPublicBoardRpc(BoardSnapshot board) { /* 場の情報は両者に送信してよい */ }
}
```

### 18-3. Unity Relayによる接続フロー（join codeベース・マッチメイキングなし）

ロビー一覧やマッチメイキングは作らず、**合言葉（join code）を口頭やメッセージで伝える**シンプルな方式にする（友人と遊ぶ規模であれば十分）。

```
[ホスト側]
1. Unity Authentication で匿名サインイン
2. RelayService.Instance.CreateAllocationAsync(maxConnections: 1)
3. Relay.Instance.GetJoinCodeAsync(allocation) → joinCodeを画面に表示
4. UnityTransportにRelayサーバー情報を設定
5. NetworkManager.Singleton.StartHost()

[参加側]
1. Unity Authentication で匿名サインイン
2. 表示されたjoinCodeを入力
3. RelayService.Instance.JoinAllocationAsync(joinCode)
4. UnityTransportにRelayサーバー情報を設定
5. NetworkManager.Singleton.StartClient()
```

必要なUnity Gaming Servicesの設定：Unity Dashboardでプロジェクトをリンクし、Authentication・Relayサービスを有効化しておく必要がある（Claude Codeに依頼する前に、Unity Dashboard側の設定が必要な旨は把握しておくとよい）。

### 18-4. 【v2予定】写し身データとプライバシー

v1では守護者が固定カードのみなので本節は対象外（送るべき写真データ自体が存在しない）。

> 写真機能を追加した際は、オンライン対戦時に**写真そのものはネットワークに一切送信しない**設計にする。各プレイヤーは自分の端末上でローカルに写し身を生成（15章、AI命名含む）し、結果（真名・技名・ステータス）だけをホストに送って遺構デッキに合流させる。写真は常に本人の端末内に留まる。

### 18-5. 切断時の挙動（v1の割り切り）

v1では再接続・再開機能は実装しない。対戦中にいずれかが切断した場合、両クライアントに「対戦相手との接続が切れました」を表示し、対戦を終了する。再接続対応は将来の拡張とする（21章）。

## 21. 今後の拡張余地（メモ）

- **【v2・最優先】写真からの守護者自動生成（15章）と、PC版の音声詠唱（16-5章）を実装する。** v1のコア対戦ロジック・AI対戦・オンライン対戦が安定して動いてから着手する。
- **上級モード「真名秘匿」**：写し身を確定する際に真名を本人にも見せず、実際にそのカードが場に出た瞬間に初めて真名が表示される（記憶を思い出す瞬間を表現する高難度ルール）。v2の写真機能が入った後の追加検討事項。
- 礎石カードの「ゾーン奪取」効果などを追加すれば、より対話的な駆け引きが生まれる。
- 写真の枚数（現在15枚）やデッキ合計枚数（現在48枚）は、実際にプレイして調整する前提の初期値。
- 系統（焔/森/流/光/影）同士の相性（例：焔は森に技ダメージ+1など）を追加すると、写真選びの戦略性がさらに増す拡張余地がある。
- AI対戦の難易度分け（17-4章）、オンライン対戦の再接続対応（18-5章）は、v1では見送り、プレイテスト後に必要性を判断する。
